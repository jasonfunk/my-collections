### 3. Monorepo root scaffolded

**Files created:**
- `package.json`
- `turbo.json`
- `tsconfig.base.json`
- `.gitignore`
- `.nvmrc`

#### `package.json` ‚Äî workspace root
```json
"workspaces": ["packages/*"]
```
Declares this as an **npm workspace** ‚Äî a monorepo that manages multiple packages (`shared`, `api`, `web`, `mobile`) from a single root. Running `npm install` at the root installs dependencies for all packages at once. The `scripts` block delegates `dev`, `build`, `lint`, and `test` to Turborepo.

#### `turbo.json` ‚Äî build pipeline
Turborepo is a build system for monorepos. It understands the dependency order between packages and runs tasks efficiently:
- `build` has `"dependsOn": ["^build"]` ‚Äî the `^` means "build my dependencies first." So `shared` always builds before `api`, `web`, and `mobile` (which all depend on it).
- `dev` is marked `persistent: true` (long-running process) and `cache: false` (never cache a dev server).
- Results are cached ‚Äî if nothing changed in a package, Turborepo skips rebuilding it.

#### `tsconfig.base.json` ‚Äî shared TypeScript config
All four packages extend this file so TypeScript settings stay consistent across the monorepo. Key settings:
- `"strict": true` ‚Äî enables all strict type checks. Think of this as the TypeScript equivalent of turning on all compiler warnings in Java/C#.
- `"target": "ES2020"` ‚Äî the JavaScript version the TypeScript compiler outputs.
- `"moduleResolution": "bundler"` ‚Äî tells TypeScript how to resolve `import` statements (modern approach used by Vite, Expo, etc.).

#### `.gitignore`
Tells git which files and folders to never track. Key exclusions:
- `node_modules/` ‚Äî npm dependencies (never commit these; they're installed from `package.json`)
- `dist/` and `build/` ‚Äî compiled output (regenerated by the build command)
- `.env` ‚Äî environment variables that may contain secrets
- `.expo/` and `ios/`/`android/` ‚Äî Expo-generated native build artifacts

#### `.nvmrc`
Specifies Node.js version `24` for this project. If you use [nvm](https://github.com/nvm-sh/nvm) (Node Version Manager), running `nvm use` in this directory automatically switches to the correct Node version.

### Key decisions made this session

| Decision | Chosen | Alternative | Reason |
|---|---|---|---|
| React architecture | SPA (Vite) | SSR (Next.js) | Dreamhost shared hosting doesn't support persistent Node.js processes needed for SSR |
| Mobile approach | Expo (React Native) | Native Android (Kotlin) | Single codebase targets both Android and iOS; Expo simplifies native feature access |
| Backend framework | NestJS | Express | NestJS uses decorators and DI patterns familiar from Java/C# (Spring Boot, ASP.NET Core) |
| Monorepo tooling | Turborepo | nx, plain scripts | Handles build ordering and caching with minimal config |
| Repo structure | Monorepo | Separate repos | Shared TypeScript types can be imported across API, web, and mobile from one place |

### 2. npm audit ‚Äî vulnerability review

**Command:**
```bash

```

**Result:** 44 vulnerabilities (11 low, 21 moderate, 12 high).

**Important context:** All 44 vulnerabilities are in **dev tooling only** ‚Äî none are in the running application (API, web, or mobile). Breakdown:

| Vulnerability | Location | Severity | Runtime impact |
|---|---|---|---|
| `ajv` ReDoS | `@nestjs/cli` (code generator) | Moderate | None ‚Äî CLI only |
| `webpack` SSRF | `@nestjs/cli` | High | None ‚Äî CLI only |
| `esbuild` dev server | `vite` dev server | Moderate | None ‚Äî local dev only |
| `fast-xml-parser` entity expansion | `react-native` CLI | High | None ‚Äî build tool only |
| `@tootallnate/once` | `jest-expo` (test runner) | Low | None ‚Äî tests only |

**Decision:** Accept all 44 as known risk. No production exposure. Revisit when underlying packages release fixes.

---

### 3. npm audit fix ‚Äî blocked by npm v11 platform issue

**Command:**
```bash
npm audit fix
```

**Result:** Failed with `EBADPLATFORM` error.

**What happened:** `npm audit fix` attempted to upgrade `lightningcss` from `1.19.0` ‚Üí `1.32.0` as part of fixing the `ajv`/`webpack` issues in the NestJS CLI chain. The new version of `lightningcss` ships platform-specific binary packages for every OS including Android (`lightningcss-android-arm64`). npm v11 attempted to install the Android binary on macOS (darwin/arm64) and failed.

This is a behavior change in npm v11 around optional platform-specific packages ‚Äî it is not a project configuration error.

**Decision:** Do not attempt `npm audit fix --force` (would apply breaking changes across React Native and Vite). Accept current state. Track as a known issue. When `lightningcss` or npm releases a fix for this behavior, revisit.
