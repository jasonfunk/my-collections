### 3. Monorepo root scaffolded

**Files created:**
- `package.json`
- `turbo.json`
- `tsconfig.base.json`

#### `package.json` ŌĆö workspace root
```json
"workspaces": ["packages/*"]
```
Declares this as an **npm workspace** ŌĆö a monorepo that manages multiple packages (`shared`, `api`, `web`, `mobile`) from a single root. Running `npm install` at the root installs dependencies for all packages at once. The `scripts` block delegates `dev`, `build`, `lint`, and `test` to Turborepo.

#### `turbo.json` ŌĆö build pipeline
Turborepo is a build system for monorepos. It understands the dependency order between packages and runs tasks efficiently:
- `build` has `"dependsOn": ["^build"]` ŌĆö the `^` means "build my dependencies first." So `shared` always builds before `api`, `web`, and `mobile` (which all depend on it).
- `dev` is marked `persistent: true` (long-running process) and `cache: false` (never cache a dev server).
- Results are cached ŌĆö if nothing changed in a package, Turborepo skips rebuilding it.

#### `tsconfig.base.json` ŌĆö shared TypeScript config
All four packages extend this file so TypeScript settings stay consistent across the monorepo. Key settings:
- `"strict": true` ŌĆö enables all strict type checks. Think of this as the TypeScript equivalent of turning on all compiler warnings in Java/C#.
- `"target": "ES2020"` ŌĆö the JavaScript version the TypeScript compiler outputs.
- `"moduleResolution": "bundler"` ŌĆö tells TypeScript how to resolve `import` statements (modern approach used by Vite, Expo, etc.).

#### `.gitignore`
Tells git which files and folders to never track. Key exclusions:
- `node_modules/` ŌĆö npm dependencies (never commit these; they're installed from `package.json`)
- `dist/` and `build/` ŌĆö compiled output (regenerated by the build command)
- `.env` ŌĆö environment variables that may contain secrets
- `.expo/` and `ios/`/`android/` ŌĆö Expo-generated native build artifacts

#### `.nvmrc`
Specifies Node.js version `24` for this project. If you use [nvm](https://github.com/nvm-sh/nvm) (Node Version Manager), running `nvm use` in this directory automatically switches to the correct Node version.

### 4. TypeORM configured in NestJS

**Two separate configs were created ŌĆö this is an important pattern:**

#### `packages/api/src/data-source.ts` ŌĆö for the TypeORM CLI
The TypeORM CLI (`typeorm-ts-node-commonjs`) runs outside of NestJS. It doesn't know about the DI container or `ConfigService`. It needs a plain `DataSource` export that reads env vars directly via `dotenv`.

```typescript
import 'reflect-metadata';
import { DataSource } from 'typeorm';
import * as dotenv from 'dotenv';

dotenv.config();

export default new DataSource({
  type: 'postgres',
  url: process.env.DATABASE_URL,
  entities: ['src/**/*.entity.ts'],
  migrations: ['src/migrations/*.ts'],
});
```

#### `packages/api/src/app.module.ts` ŌĆö for the NestJS runtime
`TypeOrmModule.forRootAsync()` integrates with the DI container. The `useFactory` function receives `ConfigService` (injected) and returns the TypeORM options. This is the correct NestJS pattern.

```typescript
TypeOrmModule.forRootAsync({
  imports: [ConfigModule],
  inject: [ConfigService],
  useFactory: (config: ConfigService) => ({
    type: 'postgres',
    url: config.get<string>('DATABASE_URL'),
    entities: [__dirname + '/**/*.entity{.ts,.js}'],
    migrations: [__dirname + '/migrations/*{.ts,.js}'],
    synchronize: false,   // NEVER true ŌĆö would auto-modify the schema on startup
    migrationsRun: true,  // runs pending migrations when the app boots
    logging: process.env.NODE_ENV !== 'production',
  }),
}),
```

**`forRoot` vs `forRootAsync`:** `forRoot` accepts a plain config object. `forRootAsync` accepts a factory function that the DI container calls after injecting dependencies ŌĆö so `ConfigService` (which requires the DI container to be ready) is available. Always use `forRootAsync` when the config depends on other services.

**`ts-node` installed** as a devDependency in the API package ŌĆö required by `typeorm-ts-node-commonjs` to execute TypeScript migration files without pre-compiling.

### 5. Migration scripts added to `packages/api/package.json`

"migration:create":   "typeorm migration:create",

**`migration:create` vs `migration:generate`:**

### 1. Conceptual foundation established

Before writing code, we mapped out the four OAuth2 roles in the context of this project:

| Role | Who it is |
|---|---|
| Resource Owner | The user (you, or another collector) |
| Client | The web app or mobile app |
| Authorization Server | NestJS `/auth/*` endpoints ŌĆö authenticates users and issues tokens |
| Resource Server | NestJS all other endpoints ŌĆö validates tokens and serves data |

The Authorization Server and Resource Server are the same NestJS process ŌĆö normal for first-party apps.

**PKCE** (Proof Key for Code Exchange) replaces client secrets for public clients (web SPA, mobile app). The client generates a random `code_verifier`, hashes it to a `code_challenge`, sends the challenge upfront, then proves it started the flow by revealing the `code_verifier` during token exchange. An attacker who intercepts the authorization code can't use it without the verifier.

**JWT access tokens** (15 min) are self-contained ŌĆö the resource server validates them by checking the signature, no DB lookup on every request.

**Refresh token rotation** (30 day tokens stored as SHA-256 hashes in DB): using a refresh token revokes it and issues a new one. Token reuse (e.g., attacker uses a stolen token) triggers revocation of all tokens for that user/client.

### 4. Entities created

Four TypeORM entities, each mapping to a PostgreSQL table:

- `user.entity.ts` ŌåÆ `users` table (id, email, passwordHash, isApproved, createdAt, updatedAt)
- `oauth-client.entity.ts` ŌåÆ `oauth_clients` table (id, clientId, clientSecretHash, name, redirectUris[], allowedScopes[])
- `authorization-code.entity.ts` ŌåÆ `authorization_codes` table (id, codeHash, userId FK, clientId FK, redirectUri, scopes[], codeChallenge, codeChallengeMethod, expiresAt, usedAt, createdAt)
- `refresh-token.entity.ts` ŌåÆ `refresh_tokens` table (id, tokenHash, userId FK, clientId FK, scopes[], expiresAt, revokedAt, createdAt)

**Key design decisions:**
- `codeHash`/`tokenHash`: raw codes/tokens are never stored ŌĆö only their SHA-256 hashes. If the DB is compromised, hashes are useless without the raw values.
- `clientSecretHash = null` marks public clients (web SPA, mobile app). Public clients use PKCE instead of client secrets.
- `usedAt`/`revokedAt` as nullable timestamps allow single-use enforcement and revocation tracking.

**TypeScript strict mode gotcha:** All entity properties need the definite assignment assertion (`!`) suffix ŌĆö TypeScript strict mode requires initialization guarantees, but TypeORM populates properties at runtime via reflection, not constructors. Pattern: `id!: string` instead of `id: string`.

## 2. Scraper: `scripts/scrape-star-wars-catalog.ts`

**Source:** `https://www.transformerland.com/wiki/star-wars/original-kenner-series/`

**Key design decisions:**

**Cloudflare managed challenge** ŌĆö plain `fetch`, `curl`, and headless Playwright (`headless: true`) all fail with Cloudflare's bot detection (TLS fingerprinting + JS challenge). Only `chromium.launch({ headless: false })` passes. A browser window opens during the ~3-minute run; this is expected and noted in the script's header comment.

After `page.goto()` with `waitUntil: 'domcontentloaded'`, wait for the CF challenge to resolve:
```typescript
await page.waitForFunction(
  () => !document.title.includes('Just a moment'),
  { timeout: 15_000 },
);
```

**Category extraction** ŌĆö embedded in the URL slug: `/wiki/toy-info/star-wars-original-kenner-series-{category-slug}-{item-slug}/{id}/`. Matched against 10 known slugs ŌåÆ `StarWarsCategory` enum value.

**figureSize** ŌĆö site's size field is unreliable (shows 0cm for figures). Hardcoded from category: `BASIC_FIGURE ŌåÆ "3.75"`, `TWELVE_INCH ŌåÆ "12"`, everything else null.

**Year** ŌĆö extracted from the "Year:" row in the info table. All 199 items have a year (1977ŌĆō1985).

**Accessories** ŌĆö from "Set Accessories" section. Strip trailing `(xN)` quantity suffix; keep color/variant descriptors (e.g., `"DL-44 Blaster Pistol (bluish black)"`).

**Error handling** ŌĆö log and continue. One transient network error (Boba Fett basic figure) was re-scraped and manually appended.

**Rate limiting** ŌĆö 1100ms sleep between requests.

**Output fields per item:**
```

isVariant, variantDescription, catalogImageUrl, sourceUrl, kennerItemNumber, year
```

**Root cause investigation:**
- Global search (`GET /collections/search`) was crashing the frontend (blank screen) because:

- Star Wars catalog page search was silently returning all results ŌĆö `CatalogBrowseQueryDto` had no `search` field, and `StarWarsCatalogService.findAll()` never applied it.

**Fixes:**
- `packages/shared/src/types/common.ts`: Made `condition` and `packagingCondition` optional (`condition?: ConditionGrade`). Nullable DB columns should flow through as optional in shared types.
- `collections-search.service.ts`: Changed `condition: item.condition` ŌåÆ `condition: item.condition ?? undefined` (and same for `packagingCondition`) to satisfy the now-optional type.
- `star-wars-catalog.dto.ts`: Added `search?: string` to `CatalogBrowseQueryDto`.

| Lesson | Detail |

### 1. Playwright site exploration (transformerland.com)

- **Faction:** `Alliance:` row in info table (not derivable reliably from URL slug ŌĆö factions like "action-masters", "bases", "other" have mixed membership)
- **Combiner detection:** `Subgroup:` info table row = `"Combiners"` (reliable)

### 2. Scraper (`scripts/scrape-transformers-catalog.ts`)

New file modeled on `scripts/scrape-he-man-catalog.ts`.

Key design decisions:
- `headless: false` required ŌĆö Cloudflare managed challenge blocks headless Playwright
- H1 parsing: `h1.replace(/^G1\s*/, '')` (zero-or-more whitespace) then `lastIndexOf(':')` to split subgroup/name
- Faction: reads `Alliance:` info table row ŌåÆ `"Autobot"` ŌåÆ `AUTOBOT`, `"Decepticon"` ŌåÆ `DECEPTICON`
- Year ŌåÆ line mapping: `1984ŌåÆG1_S1`, `1985ŌåÆG1_S2`, `1986ŌåÆG1_S3`, `1987ŌåÆG1_S4`, `1988ŌåÆG1_S5`, `1989+ŌåÆG1_S6`
- `isCombiner`: `subgroupRow === 'Combiners'`
- `combinerTeam`: H1 subgroup value when `isCombiner` (e.g. `"Aerialbots (Superion, G1)"`)
- `isMailaway`: `url.includes('mail-away')`
- Rate limit: 1100ms between requests

**Bug found and fixed:** H1 text nodes concatenated as `"G1Aerialbots..."` (no space). The original regex `/^G1\s+/` required whitespace and silently failed. Fixed to `/^G1\s*/`. The already-generated JSON was patched with a Python one-liner to strip the leading `"G1"` from all 63 `combinerTeam` values.
